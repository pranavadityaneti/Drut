1. Positioning — Practice vs Sprint

Two distinct surfaces with two distinct jobs. Keep them mentally and in-copy separate.

SurfaceJobFeelScoring
PracticeLearn & master question patterns. Safe space to attempt, fail, and prove you've got it.The gym — no penalty for trying hard things.+2 / −1, mastery via Prove It
SprintSpeed training — answer good questions fast against a timer. Volume + accuracy under time pressure.A timed rapid-fire round.flat +1 / −1
Identity (non-negotiable): Drut is a practice / preparation platform with just-in-time concept anchoring — NOT a learning-content / lessons platform. We never ship a browsable "Lessons" tab, never gate practice behind "complete this lesson," never write chapter-length explanations. Concept help appears only reactively, scoped to one pattern, inside the practice flow. "We explain the concept for this stuck moment — we don't teach the syllabus."
Correction logged: Sprint is not an exam simulation and its scoring must not mirror a real exam's marking scheme. It is a speed-training drill. The current code (sprintService) wrongly applies per-exam negative marking (JEE −1, EAPCET none, CAT 3:1) — that is exam-sim logic and must be replaced with flat +1/−1 (see §8).

2. Scoring model — three separate layers

Keep these three concepts cleanly separate in code and in the UI. They were getting conflated under the word "streak"; they are different things.

Accuracy is rewarded 2:1 in Practice (a wrong answer costs half a correct), which keeps attempting worthwhile while gently discouraging blind guessing. Sprint is flat +1/−1 — the timer + volume ("the more you answer, the more you get") is the real pressure.

What points add up to (the "tail")

  • (a) Session score — starts at 0 each session, shown on the session-summary screen.
  • (b) Lifetime XP → levels — every session's net points roll into a cumulative XP total on the profile, which drives a level. Pairs naturally with the daily streak as the long-term return loop.
  • (c) Leaderboard — deferred to post-beta.
Points ≠ mastery. Points are the engagement score. Passing Prove It (2/3) is the learning signal that actually clears a pattern off Focus Areas. Two different jobs; show them differently. Don't let a high score imply mastery, or vice-versa.

Small decision to confirm during build: does the session score floor at 0 (never shows negative) or can it go negative? Recommendation: floor the displayed session score at 0 to avoid demoralising early learners; XP accrues from net positive sessions only.

3. Streaks — consistent across Practice + Sprint

Two streaks, defined once in @drut/shared, used identically on web, Android, and iOS (per CLAUDE.md — one definition, no drift).

Answer streak (combo, in-session)

  • +1 to the combo on each correct answer; resets to 0 on a wrong answer.
  • Hitting a multiple of 5 (5, 10, 15…) awards +2 bonus points.
  • Identical logic + an identical small "🔥 ×N" indicator in both Practice and Sprint.
  • Track best_answer_streak as a personal record.
  • Prove-It correct answers extend the combo; a Prove-It wrong answer resets it (natural).

Daily streak (the flame)

  • +1 the first time each day the student does any qualifying activity — answers ≥1 Practice question or completes a Sprint. One unified counter and flame.
  • Stored at user level: last_active_date, current_streak, longest_streak.
  • Both Practice-submit and Sprint-complete call one shared recordDailyActivity() in @drut/shared.
  • Missed day → resets. (Post-beta: optional streak-freeze / grace day.)
Negative marking never touches either streak. Streaks are purely correct/incorrect + daily activity. Point deductions live only in the Points layer. No double-penalty system.

4. Prove It — the mastery drill

The mechanism that answers "has the student actually learned this pattern?"

Head (trigger) → student answers a question wrong (or correct-but-slow) → reads the solution: Quick Method (T.A.R.) + Full Solution (D.E.E.P.) → taps "Prove It"

Body → 3 fresh questions of the same pattern (fsm_tag), one at a time, each timed. Scored +2 / −1 like normal practice.

Tail (grade) → 2 of 3 correct = PASS → pattern clears off Focus Areas, mastery bumps, "Pattern cleared ✓". On the next question the student flows on normally.
    < 2 of 3 = not yet → show the Concept Card (§6), pattern stays in Focus Areas, offer "Try once more". No hard lock (§9).
ParameterValue
Drill size (N)3 questions
Pass threshold2 of 3 correct
What decides passCorrectness. Speed is a bonus badge only ("nailed it fast"), never a pass gate.
Wrong-answer penalty−1 (same as practice — not −2; the 2/3 gate is the real stake, no double-punishment in remediation)
Counts against daily quota (15/day free)No. Remediation must not be rationed.
Bank-depth requirement≥ 4 questions per fsm_tag (1 seen + 3 fresh). If a pattern is too thin, Prove It is disabled for it until the bank fills.
Hard dependency on question volume: Prove It needs at least 4 questions per pattern. Thin patterns (e.g. Chemistry today, ~5 Qs total) can't run it. This ties directly to the 150-per-chapter generation work — Prove It rolls out pattern-by-pattern as depth allows.

5. Focus Areas (renamed from "debt collector")

Drop all financial metaphors (debt / credit) and all sporty language (rematch). The dashboard reminder of patterns a student hasn't passed is called "Focus Areas" — professional, positive, exam-oriented.

  • A dashboard card listing patterns/topics where Prove It hasn't been passed.
  • Framed around readiness, not guilt — e.g. "4 patterns to firm up in Chemistry."
  • One tap on a Focus Area starts its Prove-It drill.
  • Replaces the existing DebtCollector widget; the underlying "in debt" flag is renamed to a neutral "needs-review" state (no user-facing jargon).

6. Concept anchoring + the repeated-fail Concept Card

Every solution carries a concept (enforced)

The Diagnose phase of D.E.E.P. must always state the governing principle + key formula in 1–2 lines (e.g. "Newton's 2nd law: F = ma"). This becomes a required field in the generation schema — no question can ship without a concept anchor in its full solution.

Repeated fail → the Concept Card (Option A)

Trigger: the student fails Prove It (< 2/3 on the pattern). The inline anchor clearly wasn't enough, so we escalate — once, in-flow:

A single Concept Card for that one pattern containing:
  1. the governing principle + formula (the facts),
  2. one fully-worked original example (the worked-examples feature, forlater #49),
  3. a "why this trips people up" note tied to the exact wrong option they kept choosing,
then → "Try once more" (re-drill the pattern).
  • It is a card inside the practice flow — one pattern, not a navigable lesson, not a section you must complete.
  • If they fail again: it stays in Focus Areas, suggest coming back later. Never trap them (no hard lock).
The line that keeps us a practice platform: concept content appears only reactively, scoped to one pattern, in-flow. No "Lessons" tab, no "finish to continue," no chapter write-ups. A Concept Card is remediation; a lessons section would be curriculum. We stay on the remediation side.

Option B (AI "Ask why" tutor): deferred to post-beta. The natural upgrade — open a chat pre-loaded with the question + the student's wrong answers + the concept. Highest-value, biggest build; revisit after launch.

7. Legal / content guardrail (applies to all of the above)

Textbook PDFs are not ours to show, link, send, or excerpt. This is a hard constraint, not a preference.
  • No Option C. We never surface, link, or send textbook pages to students.
  • sourcePages is strictly internal — grounding/QA metadata used during generation only, never rendered in any client.
  • Concept Cards & worked examples must be 100% original expression. Facts are free to use — a principle, a formula (F = ma isn't copyrightable) — and we write our own explanation and our own example problems. We must never reproduce a textbook's wording or its specific solved examples.
  • This is enforced in two places: the generation prompt (original-expression rule, no copied passages/solved problems) and the human review pass.

Caveat: not legal advice. This is the safe-by-design approach; recommend a quick legal review of the content pipeline ("generated-from but not reproducing") before public launch.

8. Sprint repositioning — code + copy changes

  • Scoring: replace the per-exam negative-marking config in sprintService (JEE −1, EAPCET none, CAT 3:1) with flat +1 correct / −1 wrong for every exam.
  • Combo bonus: apply the same +2-every-5 answer-streak bonus as Practice (consistency).
  • Copy/positioning: reword Sprint UI (and any marketing line) from "exam simulation" → "train your speed / answer faster."
  • Timer stays — it's the core of the speed-training mechanic.

9. No hard topic-lock (decided)

We do not lock students into a failing topic until they pass. Reasons: it would trap students on thin-bank patterns (no questions to prove with), risks beta churn at the worst moment, and can jail a student on a faulty question. Instead: soft nudge (the prominent CTA after a miss is "Prove It"), Focus Areas resurfacing, and positive streak incentives. An opt-in "Focus Mode" (student chooses to lock themselves in) can be considered later.
