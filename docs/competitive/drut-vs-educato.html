0. The one-paragraph thesis

Educato and Drut have made opposite bets. Educato spends its AI budget on the live student experience — a named tutor, concept explanations, a real-time predictive grade — and leans on a thin bank of real past-paper questions (~8 per topic). Drut spends its AI budget on manufacturing — a rigorous, textbook-grounded, adversarially-audited question factory targeting 150 questions per chapter, each carrying a dual fast/deep solution (T.A.R. + D.E.E.P.) and deep per-attempt analytics — but currently ships zero AI in the live student loop. Drut's pedagogy-per-question and its analytics are genuinely deeper than Educato's. Educato's engagement loop and its "teach, then test" structure are genuinely better than Drut's. The two are strong in mirror-image places.

In plain English: Educato makes studying feel alive and personal but has shallow content. Drut is building deep, rigorous content but the studying experience is currently static — a student answers a question, sees a (very good) solution, and moves on, with no AI helping them in the moment and no single number telling them if they're on track. We out-build them on substance; they out-build us on the feeling of momentum. The fix is to put our AI where they put theirs — into the live experience — on top of content that's already deeper than theirs.

1. Framing: what Drut actually is (and what it isn't)

Drut is a multi-exam, multi-state competitive-exam platform. EAPCET (AP + Telangana) is the MVP beachhead — a deliberate strategic decision to prove the model on one exam before fanning out, given the time and effort each additional exam takes. EAPCET focus is not a product advantage and is never listed as a "pro" in this document. The strengths below are strengths of the approach and the architecture, which are exam-agnostic by design.

Honest caveat (this is itself a "con", expanded in §7): the architecture is exam-agnostic in shape — exams are data rows (ExamDef[] in packages/shared/src/lib/taxonomy.ts), helpers are generic, ingestion is keyed by subject, time-targets are per-exam. But the live taxonomy currently contains only ap_eapcet and ts_eapcet; the mobile app hardcodes a two-exam union type; and there is no Biology syllabus yet (so NEET isn't reachable without new content). "We're not just an EAPCET app" is true of the design and intent, and the foundation genuinely generalises — but turning that into shipped multi-exam reality is real work, not a config flip. The vision is sound; the delivery gap is the thing to close.

3.2 The generation pipeline — Drut has TWO, and only one is real

This is the most important process finding. The codebase contains two pipelines:

Pipeline A — legacy server-side AI (effectively dead)

  • generate-batch / generate-question edge functions, "Universal Brain v4.0 — RAG-grounded."
  • Routes to Gemini gemini-3-flash-preview, with "Rogue Routing" trying a RunPod-hosted PhysicsWallahAI/Aryabhata-1.0 for Math/Physics first.
  • RAG-grounded via gemini-embedding-001 (768-dim) over a textbook_chunks table (~7,200 chunks).
  • Why it's effectively dead: (1) gated off — ALLOW_LIVE_AI_FALLBACK = false; (2) the model id gemini-3-flash-preview is not a real released model and would 404 (hence the aggressive fallback to gemini-1.5-flash); (3) the RunPod endpoint is a hardcoded ephemeral IP, almost certainly offline; (4) the ~7,200 existing chunks were embedded with the now-retired text-embedding-004 while new queries use gemini-embedding-001 — a silent embedding-space mismatch that degrades retrieval quality.

Pipeline B — Claude-curated bulk import (the real one)

  • Per docs/practice-mode-architecture-v2.md, server-side generation was strategically retired for MVP in favour of out-of-band curation.
  • Process (from docs/question-generation/INSTRUCTIONS.md): Claude reads the actual textbook PDF → enumerates subtopics with page ranges → generates batches of 50 → validates with scripts/validate-bulk-import.mjs → adversarially audits every Hard question with parallel agents (independent re-derivation, distractor mistake-path check, T.A.R. genuine-vs-relabelled, PROOF completeness) → Pranav uploads via Admin Bulk Import (his JWT).
  • Ground-truth rule: "NEVER generate from memory. No PDF → no generation." Every question is grounded in a specific textbook page.
  • Difficulty split: 30% Easy / 50% Medium / 20% Hard = 45E / 75M / 30H per 150-question chapter (exactly the split you asked about earlier).
  • Distractor engineering: each of the 3 wrong options is designed to map to a specific mistake (sign error, formula confusion, arithmetic slip) — so wrong answers are diagnostic.
In plain English: Drut's "AI question factory" you may picture running on a server is essentially mothballed and partly broken. The questions that actually reach students are made by Claude reading the real textbook, generating in batches, getting the hard ones adversarially fact-checked, and then you uploading them. That's a higher-quality process than a server cranking out ungrounded questions — but it's manual and gated on you, so it doesn't scale on its own.

3.3 Verification / QA taxonomy

Drut has a genuine trust hierarchy (intended order): v3-verified-pyq > manual-curated > manual > v3-verified-textbook > v3-verified-rag > v3-unverified-ai. Educato's QA is "human-in-the-loop validation" (their words) on a much smaller, mostly-PYQ set.

Bug found during this audit (worth fixing): the Admin Bulk Import edge function stamps verification_status: 'admin-verified', but the docs, the migration, and the Bulk Import UI all expect 'manual-curated'. Critically, neither value is in the mobile "trusted question" filter (TRUSTED_STATUS_SUBSTRINGS in apps/mobile/hooks/usePracticeQuestions.ts). So your hand-curated, highest-effort questions currently fall through to client-side keyword validation instead of being trusted — the opposite of intended. Cheap to fix (add the value to the filter), but it means the best content is being treated as least-trusted on mobile right now.

5. MODEL — how each one teaches (atom level)

5.1 The pedagogical unit

Educato's unit is a Study Session: a 4–9 minute sequence that interleaves multi-part Concept cards (hook → definition → history → worked examples → matching exercise) with Question cards, ending on a predictive-grade verdict. Teach, then test, then score.

Drut's unit is a Question: the student picks exam/subject/chapter/difficulty/count and answers questions; each question carries two solution artifacts:

  • T.A.R. (Trigger → Action → Result) — a 3-step shortcut, shown only when a genuine pattern shortcut exists (theOptimalPath.exists). UI tab: "Quick Method".
  • D.E.E.P. (Diagnose → Extract → Execute → Proof) — a 4-phase full solution, always present. UI tab: "Full Solution".

The pedagogical theory (from docs/learning-framework.md) is explicit and, frankly, sophisticated: it targets two failure modes — "the plodder" (masters concepts, runs out of time) and "the pattern-matcher" (memorises shortcuts, can't adapt) — by giving every question both a fast solve and a deep solve, side by side.

This is Drut's sharpest pedagogical edge. Educato's wrong-answer screen gives a 1–2 sentence explanation per option. Drut gives a speed-optimised shortcut and a rigorous 4-phase derivation, designed to fight two distinct ways students fail an exam. Nothing in Educato matches the per-question teaching depth — provided the content is actually populated and rendered correctly.

5.3 Progress / mastery model

Drut tracks mastery at the pattern level, which is more granular than topic level. A 3-state FSM (novice → learning → verified at streak thresholds 3 and 7) is keyed on fsm_tag — the solution pattern, not the chapter. Crucially, a "success" requires correct AND fast (within per-exam timeTargets) — so speed is a first-class signal, matching how EAPCET/JEE actually punish slowness. Skipping a needed drill marks the pattern "in debt" for resurfacing. Educato tracks at the topic level (visible), feeding the predictive grade.

5.4 Analytics

This is a quiet Drut strength. Drut exposes 10 analytics RPCs: overall accuracy/speed, per-topic proficiency, learning velocity, weakest subtopics, activity heatmap, distractor analysis (which wrong answers you pick), stamina curve (accuracy vs question index), sprint performance, streak, and pattern-mastery rollup — with matching dashboard widgets. Educato's visible analytics are essentially the predictive grade + a streak. Drut has materially richer data; it just hasn't turned that data into a single emotional signal the way Educato has.

6. Pros of Drut's approach & model

These are strengths of the method and architecture — they hold across any exam we add.

  1. Deepest-in-class per-question teaching (T.A.R. + D.E.E.P.). A dual fast/deep solution per question, theory-grounded in two named failure modes. Educato has nothing comparable at the question level. This is the single biggest pedagogical differentiator.
  2. Pattern-level mastery, not topic-level. Tracking on fsm_tag (solution pattern) means we know a student has mastered "completing the square" across chapters — finer-grained than "70% in Algebra." "Prove It" then resurfaces the same pattern for targeted remediation.
  3. Speed is a first-class signal. Mastery requires correct and within an exam-specific time target. Competitive exams are speed games; modelling time at the core (per-exam timeTargets) is a genuine edge Educato doesn't show.
  4. Analytics surface is far richer. Distractor analysis, stamina curves, learning velocity, weakest-subtopic detection — 10 RPCs of signal. We can build a better predictive grade than Educato because we already collect more of the inputs.
  5. Rigorous, grounded generation process. Textbook-PDF-grounded ("no PDF, no generation"), page-level provenance (sourcePages), batch validation, and adversarial audit of every Hard question. This is a more defensible quality bar than ungrounded generation — and it's powered by a frontier model (Claude), not a cheap one.
  6. Generative depth is a real content moat — once filled. 150/chapter design vs Educato's ~8/topic is 15–20× depth. At scale, Drut's Physics alone would exceed Educato's entire 4-subject bank. Depth lets us offer endless fresh practice and isomorphic variants; thin PYQ banks get exhausted.
  7. Distractors are engineered, not random. Each wrong option encodes a specific misconception at generation time. That's the raw material for instant, precise "here's the mistake you made" feedback — cheaper and faster than Educato's because it's pre-computed.
  8. Exam-agnostic foundation. Exams are data (ExamDef[]), helpers are generic, ingestion is subject-keyed, time-targets are per-exam, and there's class-split scaffolding (EXAM_SYLLABUS_CONFIG). Adding an exam is fundamentally "add rows + ingest textbooks," not a rewrite. This is what makes the multi-exam vision buildable rather than aspirational.
  9. Visual system is clean and extensible. A 3-way discriminated union (none / sanitized inline svg / smiles via RDKit) covers graphs, geometry, circuits, ray diagrams, and organic chemistry — generated inline, no image hosting. Educato's questions are largely text.

7. Cons of Drut's approach & model (where to improve)

Ordered roughly by impact on the student experience.

  1. No AI tutor — the biggest experiential gap. At the moment of confusion, the Drut student is alone with a (good) static solution. Educato's Plumi answers "why is C right, not B?" with a structured, context-aware explanation. We already have everything a tutor needs as context (the question, the chosen wrong answer, the engineered distractor, the D.E.E.P. solution) — we just don't expose a conversation.
  2. No predictive grade / readiness score. Educato's entire retention loop runs on one moving number. We have richer analytics but no single "where do you stand / are you ready" signal. Highest-ROI feature gap, and unusually cheap for us because the data already exists.
  3. No concept / lesson layer. Drut is question-only. Educato teaches the concept first, then tests. A beachhead full of crammers benefits from a "learn it" surface — and it's where Educato's interleaving and worked-examples shine.
  4. No spaced repetition. A TODO and a fake alert. SR is the gold-standard retention mechanism and Educato claims it. Our "debt" flag is binary resurfacing, not scheduled review.
  5. No adaptive selection. The main queue is RANDOM() after filters. We detect weak spots but don't steer practice toward them. This is a SQL/ranking change, not an ML project — low effort, high perceived intelligence.
  6. No worked examples after the solution. (Already queued, forlater #49.) Educato shows the method on 2–3 sibling problems; we show one.
  7. Web LaTeX is broken. The canonical web SolutionView renders markdown only — math doesn't render as math on web (mobile is fine via KaTeX). For a math-heavy product this is a real correctness defect, not cosmetic.
  8. The "best content treated as least-trusted" bug (see §3.3): admin-imported, hand-audited questions aren't in the mobile trusted filter. Fix before beta.
  9. Two divergent schemas + out-of-band columns. A Gemini-permissive schema and a Bulk-Import-strict schema disagree on field names (exists vs available, phases[] vs flat steps[], different timeTargets keys). The UI reads defensively across all shapes — which works but is fragile. Several columns (verification_status, source_type, source) were added out-of-band with no migration DDL.
  10. Most of the server-side AI infra is non-functional. Dead model ids, an ephemeral RunPod IP, an embedding-space mismatch on the RAG corpus, and a master kill-switch off. This isn't hurting students (the Claude path works), but it's a large surface of misleading, rotting code that will confuse future work and gives a false sense of "we have live AI generation."
  11. Thin bank today. ~2,057 cached, ~700 EAPCET, Chemistry critically thin (~5), Math thin. Our depth advantage is potential until generation fills the bank with quality. Educato's thin bank is at least 100% authentic PYQ.
  12. No social proof / crowd stats. Educato's per-option "% of students who chose this" is engaging and cheap. We collect the data (distractor analysis) but don't show it to students.
  13. No engagement game loop. Educato's energy economy drives daily return. Our streak is passive analytics; the 15/day quota is a paywall, not a game.
  14. Exam-agnostic in design, EAPCET-only in delivery. Live taxonomy = 2 exams, mobile hardcodes the two-exam type, no Biology (no NEET). To realise the multi-exam vision, the taxonomy needs to be the genuine single source of truth (retire the legacy constants.ts exam list), mobile needs to read it dynamically, and Biology content needs to exist.

8. Where to use AI smartly (the real lesson from Educato)

Educato's lesson isn't "use more AI" — it's put AI where it compounds engagement and understanding, and use cheap sources (PYQ) for the commodity (questions). Drut has done the inverse. Here's where AI earns its cost in the live loop, roughly by ROI:

  1. An "Ask why" tutor at the moment of confusion (highest ROI). Pre-load context we already have: the question, the student's wrong option, that option's engineered misconception, and the D.E.E.P. solution. The model only has to converse, not derive from scratch — which keeps it cheap and accurate. This is the single most impactful place to spend AI.
  2. AI-narrated readiness ("predictive grade", but better). We already compute distractor patterns, stamina, velocity, weak subtopics. Use a model to turn that into a short, natural-language "here's where you stand, here's the one thing to fix next" — and a moving readiness number. We can out-do Educato here because our inputs are richer.
  3. On-demand worked examples. Generate 2–3 sibling problems for a pattern, cache them. Cheap, and it's the §6 / forlater #49 feature.
  4. A concept layer, AI-authored from the textbook. If we add a "learn it" surface, generate the concept cards (hook → definition → worked examples) grounded in the same PDFs we already ingest — exactly Educato's move, on deeper content.
  5. Instant distractor feedback — no runtime AI needed. Because each wrong option already encodes its misconception at generation time, "here's the mistake you made" can be shown instantly with zero model call. Smart AI usage sometimes means pre-computing with AI so runtime is free.
  6. Adaptive selection — no LLM needed. Re-rank the practice queue by the mastery/weak-spot data we already store. "Smart" here is a ranking change, not a model call.
Model strategy (re: your OpenAI key). The right pattern is right model per job, not one model everywhere (today everything routes to a single Gemini id that may be broken):
  • Generation + adversarial audit → frontier model (Claude) — already how Pipeline B works; keep it.
  • Live tutor at scale → cost-optimised tier (e.g. GPT-4o-mini / Gemini Flash / a small Claude) because context is pre-loaded. Your OpenAI key is exactly the lever to A/B GPT vs Gemini vs Claude on tutor quality-per-rupee before committing.
  • Readiness narration → mid-tier, low volume (once per session).
  • Embeddings → fix the mismatch: re-embed the corpus with one current model so queries and chunks share a space.
Security note: the key belongs only in a server-side env var / Supabase secret — never in client code, never committed. (And the key shared in chat should still be rotated whenever convenient; not urgent, but free.)
